# Requirements Document

## Introduction

The Game Service is a bounded context within the fullstack crash game application responsible for managing the complete game lifecycle, including rounds, bets, multiplier progression, crash points, and real-time communication via WebSocket. It ensures provably fair gameplay through cryptographic verification, enforces betting rules and constraints, and integrates with the Wallet Service via asynchronous messaging for monetary operations. The service exposes REST endpoints for game state retrieval and player actions, while broadcasting real-time updates to all connected clients via WebSocket.

## Glossary

- **Game_Service**: The microservice responsible for managing crash game rounds, bets, and real-time gameplay
- **Round**: A complete game cycle consisting of betting phase, running phase, and crash
- **Betting_Phase**: The time window during which players can place bets for the upcoming round
- **Running_Phase**: The period when the multiplier is actively increasing and players can cash out
- **Multiplier**: A numeric value starting at 1.00x that increases continuously during the running phase
- **Crash_Point**: The pre-determined multiplier value at which the round ends
- **Bet**: A wager placed by a player during the betting phase with a specific amount
- **Cash_Out**: The action of a player exiting a round at the current multiplier to secure winnings
- **Payout**: The amount won by a player, calculated as bet amount × cash out multiplier
- **Player**: An authenticated user identified by a unique player ID from the JWT_Token
- **Centavo**: The smallest monetary unit, representing 1/100 of the base currency
- **Provably_Fair_Algorithm**: Cryptographic system ensuring crash points are pre-determined and verifiable
- **Server_Seed**: A random value generated by the server used to calculate the crash point
- **Server_Seed_Hash**: SHA-256 hash of the server seed, revealed before the round starts
- **Client_Seed**: An optional random value provided by players to influence crash point calculation
- **Hash_Chain**: A sequence of cryptographically linked server seeds ensuring integrity
- **Wallet_Service**: The bounded context responsible for managing player wallets and balances
- **Message_Broker**: RabbitMQ instance used for asynchronous communication between services
- **JWT_Token**: JSON Web Token issued by Keycloak containing player authentication and identification
- **API_Gateway**: Kong gateway that routes requests and validates JWT tokens
- **WebSocket_Gateway**: Infrastructure component managing WebSocket connections and event broadcasting
- **Round_Repository**: Infrastructure component responsible for persisting round data to PostgreSQL
- **Bet_Repository**: Infrastructure component responsible for persisting bet data to PostgreSQL
- **Bet_Placed_Event**: Message published to Wallet Service when a player places a bet
- **Cashout_Event**: Message published to Wallet Service when a player cashes out
- **Bet_Lost_Event**: Message published to Wallet Service when a player's bet is lost
- **Insufficient_Balance_Event**: Message received from Wallet Service when a bet is rejected due to insufficient funds
- **Round_State**: The current status of a round (Betting, Running, Crashed, Finished)
- **Bet_State**: The current status of a bet (Pending, Active, CashedOut, Lost)
- **Concurrent_Operation**: Multiple operations attempting to modify the same round or bet simultaneously
- **House_Edge**: The statistical advantage built into the crash point calculation (e.g., 3%)

## Requirements

### Requirement 1: Round Creation and Initialization

**User Story:** As the Game Service, I want to create new rounds automatically, so that the game operates continuously without manual intervention.

#### Acceptance Criteria

1. WHEN the Game_Service starts, THE Game_Service SHALL create an initial Round in Betting_Phase
2. WHEN a Round transitions to Finished state, THE Game_Service SHALL create a new Round in Betting_Phase
3. WHEN a Round is created, THE Game_Service SHALL generate a Server_Seed using a cryptographically secure random number generator
4. WHEN a Server_Seed is generated, THE Game_Service SHALL compute the Server_Seed_Hash using SHA-256
5. WHEN a Round is created, THE Game_Service SHALL calculate the Crash_Point using the Provably_Fair_Algorithm
6. WHEN a Round is created, THE Game_Service SHALL persist the Round to the Round_Repository with Server_Seed, Server_Seed_Hash, Crash_Point, and timestamps
7. WHEN a Round is created, THE Game_Service SHALL broadcast the Server_Seed_Hash to all connected WebSocket clients

### Requirement 2: Betting Phase Management

**User Story:** As a player, I want a dedicated time window to place my bet, so that I can participate in the upcoming round.

#### Acceptance Criteria

1. WHEN a Round enters Betting_Phase, THE Game_Service SHALL start a configurable countdown timer (default 10 seconds)
2. WHILE a Round is in Betting_Phase, THE Game_Service SHALL accept bet placement requests from authenticated players
3. WHEN the Betting_Phase timer expires, THE Game_Service SHALL transition the Round to Running_Phase
4. WHEN a Round transitions from Betting_Phase to Running_Phase, THE Game_Service SHALL broadcast a round start event to all connected WebSocket clients
5. WHEN a Round transitions to Running_Phase, THE Game_Service SHALL transition all Pending bets to Active state
6. THE Game_Service SHALL broadcast the remaining Betting_Phase time to all connected WebSocket clients every second

### Requirement 3: Bet Placement

**User Story:** As a player, I want to place a bet during the betting phase, so that I can participate in the crash game round.

#### Acceptance Criteria

1. WHEN an authenticated player sends a POST request to /games/bet with a bet amount, THE Game_Service SHALL validate that the current Round is in Betting_Phase
2. WHEN a bet placement request is received, THE Game_Service SHALL validate that the bet amount is between 100 centavos (1.00) and 100000 centavos (1000.00)
3. WHEN a bet placement request is received, THE Game_Service SHALL validate that the player has not already placed a bet in the current Round
4. WHEN a valid bet is received, THE Game_Service SHALL create a Bet entity with Pending state associated with the player and current Round
5. WHEN a Bet is created, THE Game_Service SHALL persist the Bet to the Bet_Repository
6. WHEN a Bet is persisted, THE Game_Service SHALL publish a Bet_Placed_Event to the Message_Broker with player ID and bet amount in centavos
7. WHEN a Bet is successfully placed, THE Game_Service SHALL return the Bet ID and status to the player
8. WHEN a Bet is successfully placed, THE Game_Service SHALL broadcast the bet information to all connected WebSocket clients
9. IF a player attempts to bet during Running_Phase or Crashed state, THEN THE Game_Service SHALL reject the bet with an error indicating betting is closed
10. IF a player attempts to place a second bet in the same Round, THEN THE Game_Service SHALL reject the bet with an error indicating one bet per round limit

### Requirement 4: Multiplier Progression

**User Story:** As a player, I want to see the multiplier increase in real-time, so that I can decide when to cash out.

#### Acceptance Criteria

1. WHEN a Round transitions to Running_Phase, THE Game_Service SHALL initialize the Multiplier at 1.00x
2. WHILE a Round is in Running_Phase, THE Game_Service SHALL increase the Multiplier continuously using a deterministic formula
3. THE Game_Service SHALL calculate Multiplier values using exact arithmetic to ensure consistency across all clients
4. THE Game_Service SHALL broadcast Multiplier updates to all connected WebSocket clients at a minimum frequency of 10 times per second
5. WHEN the Multiplier reaches the Crash_Point, THE Game_Service SHALL transition the Round to Crashed state
6. THE Multiplier progression formula SHALL be deterministic and reproducible given the same start time and crash point

### Requirement 5: Cash Out Processing

**User Story:** As a player, I want to cash out at the current multiplier, so that I can secure my winnings before the crash.

#### Acceptance Criteria

1. WHEN an authenticated player sends a POST request to /games/bet/cashout, THE Game_Service SHALL validate that the current Round is in Running_Phase
2. WHEN a cash out request is received, THE Game_Service SHALL validate that the player has an Active bet in the current Round
3. WHEN a cash out request is received, THE Game_Service SHALL validate that the player has not already cashed out
4. WHEN a valid cash out request is received, THE Game_Service SHALL capture the current Multiplier value
5. WHEN the Multiplier is captured, THE Game_Service SHALL calculate the Payout as bet amount × current Multiplier in centavos
6. WHEN the Payout is calculated, THE Game_Service SHALL transition the Bet to CashedOut state and store the cash out Multiplier and Payout
7. WHEN a Bet transitions to CashedOut state, THE Game_Service SHALL persist the updated Bet to the Bet_Repository
8. WHEN a Bet is persisted, THE Game_Service SHALL publish a Cashout_Event to the Message_Broker with player ID, bet amount, Multiplier, and Payout in centavos
9. WHEN a cash out is successful, THE Game_Service SHALL return the Payout amount and Multiplier to the player
10. WHEN a cash out is successful, THE Game_Service SHALL broadcast the cash out information to all connected WebSocket clients
11. IF a player attempts to cash out during Betting_Phase or after Crashed state, THEN THE Game_Service SHALL reject the request with an error
12. IF a player attempts to cash out without an Active bet, THEN THE Game_Service SHALL reject the request with an error

### Requirement 6: Round Crash and Completion

**User Story:** As the Game Service, I want to end rounds at the pre-determined crash point, so that the game follows provably fair rules.

#### Acceptance Criteria

1. WHEN the Multiplier reaches the Crash_Point, THE Game_Service SHALL transition the Round to Crashed state
2. WHEN a Round transitions to Crashed state, THE Game_Service SHALL identify all Active bets that have not cashed out
3. WHEN Active bets are identified, THE Game_Service SHALL transition those bets to Lost state
4. WHEN bets transition to Lost state, THE Game_Service SHALL persist the updated bets to the Bet_Repository
5. WHEN bets are persisted, THE Game_Service SHALL publish Bet_Lost_Event messages to the Message_Broker for each lost bet
6. WHEN a Round transitions to Crashed state, THE Game_Service SHALL broadcast a crash event to all connected WebSocket clients with the Crash_Point and Server_Seed
7. WHEN a Round transitions to Crashed state, THE Game_Service SHALL transition the Round to Finished state after a configurable delay (default 3 seconds)
8. WHEN a Round transitions to Finished state, THE Game_Service SHALL persist the final Round state to the Round_Repository

### Requirement 7: Provably Fair Crash Point Generation

**User Story:** As a player, I want the crash point to be provably fair, so that I can trust the game is not manipulated.

#### Acceptance Criteria

1. WHEN a Round is created, THE Game_Service SHALL generate a Server_Seed using a cryptographically secure random number generator with at least 128 bits of entropy
2. WHEN a Server_Seed is generated, THE Game_Service SHALL compute the Crash_Point using HMAC-SHA256 with the Server_Seed as the key and a deterministic input
3. WHEN computing the Crash_Point, THE Game_Service SHALL apply a House_Edge of 3% to the calculation
4. WHEN computing the Crash_Point, THE Game_Service SHALL ensure the result is a decimal value with two decimal places (e.g., 1.50x, 2.37x, 10.00x)
5. THE Crash_Point calculation SHALL be deterministic such that the same Server_Seed always produces the same Crash_Point
6. THE Game_Service SHALL support optional Client_Seed values to allow players to influence crash point generation
7. WHEN a Client_Seed is provided, THE Game_Service SHALL combine it with the Server_Seed using HMAC-SHA256 before calculating the Crash_Point

### Requirement 8: Provably Fair Verification

**User Story:** As a player, I want to verify past round results, so that I can confirm the game was fair.

#### Acceptance Criteria

1. WHEN a player sends a GET request to /games/rounds/:roundId/verify, THE Game_Service SHALL retrieve the Round from the Round_Repository
2. WHEN a Round is retrieved for verification, THE Game_Service SHALL return the Server_Seed, Server_Seed_Hash, Client_Seed (if used), Crash_Point, and calculation algorithm
3. THE verification response SHALL include step-by-step instructions for independently calculating the Crash_Point from the seeds
4. THE Game_Service SHALL provide a reference implementation or pseudocode for the Crash_Point calculation in the verification response
5. IF a Round does not exist, THEN THE Game_Service SHALL return an error indicating the round was not found

### Requirement 9: Current Round State Retrieval

**User Story:** As a player, I want to view the current round state, so that I can see the multiplier, bets, and phase.

#### Acceptance Criteria

1. WHEN a player sends a GET request to /games/rounds/current, THE Game_Service SHALL retrieve the current Round from the Round_Repository
2. WHEN the current Round is retrieved, THE Game_Service SHALL return the Round ID, Round_State, current Multiplier (if running), Server_Seed_Hash, and remaining Betting_Phase time
3. WHEN the current Round is retrieved, THE Game_Service SHALL return all bets placed in the Round with player usernames, bet amounts, and bet states
4. THE Game_Service SHALL exclude the Server_Seed from the response until the Round has crashed
5. THE Game_Service SHALL return the Crash_Point only after the Round has crashed

### Requirement 10: Round History Retrieval

**User Story:** As a player, I want to view past round results, so that I can analyze game patterns and verify fairness.

#### Acceptance Criteria

1. WHEN a player sends a GET request to /games/rounds/history, THE Game_Service SHALL retrieve finished rounds from the Round_Repository in descending order by creation time
2. THE Game_Service SHALL support pagination with configurable page size (default 20, maximum 100)
3. WHEN round history is retrieved, THE Game_Service SHALL return Round ID, Crash_Point, Server_Seed_Hash, creation timestamp, and total bet count for each round
4. THE Game_Service SHALL exclude Server_Seed values from the history response to maintain hash chain integrity for future rounds
5. THE Game_Service SHALL return pagination metadata including total count, current page, and total pages

### Requirement 11: Player Bet History Retrieval

**User Story:** As a player, I want to view my bet history, so that I can track my gameplay and winnings.

#### Acceptance Criteria

1. WHEN an authenticated player sends a GET request to /games/bets/me, THE Game_Service SHALL retrieve all bets for the player from the Bet_Repository in descending order by creation time
2. THE Game_Service SHALL support pagination with configurable page size (default 20, maximum 100)
3. WHEN bet history is retrieved, THE Game_Service SHALL return Bet ID, Round ID, bet amount, Bet_State, cash out Multiplier (if cashed out), Payout (if cashed out), and timestamp for each bet
4. THE Game_Service SHALL return pagination metadata including total count, current page, and total pages
5. THE Game_Service SHALL ensure players can only retrieve their own bet history

### Requirement 12: WebSocket Connection Management

**User Story:** As a player, I want to receive real-time game updates, so that I can see the multiplier and other players' actions instantly.

#### Acceptance Criteria

1. THE Game_Service SHALL expose a WebSocket endpoint for client connections
2. WHEN a client connects to the WebSocket endpoint, THE Game_Service SHALL establish a persistent bidirectional connection
3. WHEN a WebSocket connection is established, THE Game_Service SHALL send the current Round state to the client
4. WHEN a WebSocket connection is closed, THE Game_Service SHALL clean up associated resources
5. THE Game_Service SHALL handle WebSocket connection errors gracefully and log connection failures
6. THE Game_Service SHALL support multiple concurrent WebSocket connections from different clients

### Requirement 13: Real-Time Event Broadcasting

**User Story:** As a player, I want to see other players' bets and cash outs in real-time, so that I can gauge game activity.

#### Acceptance Criteria

1. WHEN a new Round is created, THE Game_Service SHALL broadcast a round_created event to all connected WebSocket clients with Round ID and Server_Seed_Hash
2. WHEN a Round transitions to Running_Phase, THE Game_Service SHALL broadcast a round_started event to all connected WebSocket clients
3. WHILE a Round is in Running_Phase, THE Game_Service SHALL broadcast multiplier_update events to all connected WebSocket clients with the current Multiplier value
4. WHEN a player places a bet, THE Game_Service SHALL broadcast a bet_placed event to all connected WebSocket clients with player username and bet amount
5. WHEN a player cashes out, THE Game_Service SHALL broadcast a cashout event to all connected WebSocket clients with player username, Multiplier, and Payout
6. WHEN a Round crashes, THE Game_Service SHALL broadcast a round_crashed event to all connected WebSocket clients with Crash_Point and Server_Seed
7. THE Game_Service SHALL broadcast Betting_Phase countdown updates every second to all connected WebSocket clients

### Requirement 14: Wallet Service Integration

**User Story:** As the Game Service, I want to communicate with the Wallet Service asynchronously, so that player balances are updated correctly.

#### Acceptance Criteria

1. WHEN the Game_Service starts, THE Game_Service SHALL connect to the Message_Broker using the configured connection parameters
2. WHEN a Bet is placed, THE Game_Service SHALL publish a Bet_Placed_Event to the Message_Broker with event ID, player ID, bet ID, bet amount in centavos, and timestamp
3. WHEN a player cashes out, THE Game_Service SHALL publish a Cashout_Event to the Message_Broker with event ID, player ID, bet ID, Payout amount in centavos, Multiplier, and timestamp
4. WHEN a bet is lost, THE Game_Service SHALL publish a Bet_Lost_Event to the Message_Broker with event ID, player ID, bet ID, bet amount in centavos, and timestamp
5. THE Game_Service SHALL subscribe to the Insufficient_Balance_Event queue from the Wallet_Service
6. WHEN an Insufficient_Balance_Event is received, THE Game_Service SHALL mark the corresponding Bet as rejected and notify the player
7. THE Game_Service SHALL acknowledge messages to the Message_Broker only after successful processing
8. IF message processing fails, THEN THE Game_Service SHALL reject the message and allow redelivery according to the Message_Broker retry policy

### Requirement 15: Monetary Precision

**User Story:** As a system administrator, I want all monetary values to be handled with exact precision, so that no rounding errors or financial discrepancies occur.

#### Acceptance Criteria

1. THE Game_Service SHALL store all bet amounts and Payout values as integers representing centavos
2. THE Game_Service SHALL perform all monetary arithmetic operations using integer arithmetic
3. WHEN calculating Payout, THE Game_Service SHALL multiply the bet amount by the Multiplier and round down to the nearest centavo
4. THE Game_Service SHALL reject any bet amount that cannot be represented as an integer number of centavos
5. THE Bet_Repository SHALL store bet amounts and Payout values using BIGINT, NUMERIC, or equivalent exact precision database types

### Requirement 16: Concurrent Operation Safety

**User Story:** As a system administrator, I want concurrent game operations to be handled safely, so that race conditions do not corrupt game state.

#### Acceptance Criteria

1. WHEN multiple players attempt to place bets simultaneously, THE Game_Service SHALL process each bet atomically
2. WHEN multiple players attempt to cash out simultaneously, THE Game_Service SHALL process each cash out atomically with the correct Multiplier value
3. THE Game_Service SHALL use database transactions to ensure atomic updates to Round and Bet entities
4. THE Game_Service SHALL use optimistic locking or pessimistic locking to prevent concurrent modification conflicts
5. IF a Concurrent_Operation fails due to a conflict, THEN THE Game_Service SHALL retry the operation up to three times

### Requirement 17: Authentication and Authorization

**User Story:** As a player, I want my game actions to be secure, so that only I can place bets and cash out on my behalf.

#### Acceptance Criteria

1. WHEN a request is received at /games/bet, /games/bet/cashout, or /games/bets/me, THE Game_Service SHALL validate the JWT_Token in the Authorization header
2. WHEN a JWT_Token is validated, THE Game_Service SHALL extract the player ID and username from the token claims
3. IF a JWT_Token is missing or invalid, THEN THE Game_Service SHALL return an authentication error with HTTP status 401
4. WHEN placing a bet or cashing out, THE Game_Service SHALL associate the action with the authenticated player ID from the JWT_Token
5. WHEN retrieving bet history, THE Game_Service SHALL ensure the player can only retrieve their own bets

### Requirement 18: Database Persistence

**User Story:** As a system administrator, I want game data to be persisted reliably, so that round and bet information is not lost during service restarts.

#### Acceptance Criteria

1. THE Game_Service SHALL connect to the PostgreSQL games database using the configured connection parameters
2. THE Round_Repository SHALL store Round records with Round ID, Server_Seed, Server_Seed_Hash, Crash_Point, Round_State, and timestamps
3. THE Bet_Repository SHALL store Bet records with Bet ID, Round ID, player ID, bet amount, Bet_State, cash out Multiplier, Payout, and timestamps
4. WHEN a Round or Bet is created or modified, THE Game_Service SHALL persist the changes within a database transaction
5. THE Game_Service SHALL use an ORM (Prisma, MikroORM, or TypeORM) for database operations
6. THE Round_Repository SHALL define a schema with Crash_Point stored as NUMERIC type and bet amounts stored as BIGINT type

### Requirement 19: Health Check Endpoint

**User Story:** As a DevOps engineer, I want to monitor the health of the Game Service, so that I can detect and respond to service failures.

#### Acceptance Criteria

1. THE Game_Service SHALL expose a GET /health endpoint
2. WHEN the /health endpoint is accessed, THE Game_Service SHALL verify connectivity to the PostgreSQL database
3. WHEN the /health endpoint is accessed, THE Game_Service SHALL verify connectivity to the Message_Broker
4. WHEN the /health endpoint is accessed, THE Game_Service SHALL verify the WebSocket_Gateway is operational
5. WHEN all dependencies are healthy, THE Game_Service SHALL return HTTP status 200 with a success message
6. IF any dependency is unhealthy, THEN THE Game_Service SHALL return HTTP status 503 with details of the failing dependency

### Requirement 20: Error Handling and Logging

**User Story:** As a developer, I want comprehensive error handling and logging, so that I can diagnose and fix issues quickly.

#### Acceptance Criteria

1. WHEN an error occurs during bet placement or cash out, THE Game_Service SHALL log the error with the player ID, Round ID, and error details
2. WHEN an error occurs during a REST request, THE Game_Service SHALL return a structured error response with an error code and message
3. THE Game_Service SHALL log all bet placements, cash outs, and round state transitions with timestamps, player IDs, and amounts
4. THE Game_Service SHALL log connection failures to the PostgreSQL database or Message_Broker
5. WHEN a Round crashes, THE Game_Service SHALL log the Crash_Point, number of active bets, number of cashed out bets, and total Payout amount
6. THE Game_Service SHALL log WebSocket connection events (connect, disconnect, errors) with client identifiers

### Requirement 21: Domain-Driven Design Architecture

**User Story:** As a developer, I want the codebase to follow DDD principles, so that the business logic is clearly separated from infrastructure concerns.

#### Acceptance Criteria

1. THE Game_Service SHALL organize code into domain, application, infrastructure, and presentation layers
2. THE domain layer SHALL contain the Round entity with state transition methods and the Bet entity with cash out logic
3. THE domain layer SHALL contain value objects for Money, RoundId, BetId, PlayerId, and Multiplier
4. THE application layer SHALL contain use cases for round management, bet placement, cash out processing, and history retrieval
5. THE infrastructure layer SHALL contain the Round_Repository, Bet_Repository, Message_Broker client, WebSocket_Gateway, and database configuration
6. THE presentation layer SHALL contain REST controllers, WebSocket event handlers, and DTOs for API endpoints

### Requirement 22: TypeScript Strict Mode

**User Story:** As a developer, I want the codebase to use TypeScript strict mode, so that type safety is enforced and runtime errors are minimized.

#### Acceptance Criteria

1. THE Game_Service SHALL enable TypeScript strict mode in tsconfig.json
2. THE Game_Service SHALL define explicit types for all function parameters and return values
3. THE Game_Service SHALL use strict null checks to prevent null and undefined errors
4. THE Game_Service SHALL define interfaces or types for all DTOs, entities, and domain objects
5. THE Game_Service SHALL use readonly modifiers for immutable properties where appropriate

### Requirement 23: Unit Testing for Domain Layer

**User Story:** As a developer, I want comprehensive unit tests for the domain layer, so that core business logic is verified to work correctly.

#### Acceptance Criteria

1. THE Game_Service SHALL include unit tests for the Round entity state transition methods
2. THE Game_Service SHALL include unit tests for the Bet entity cash out calculation
3. THE Game_Service SHALL include unit tests verifying that bet placement is rejected outside Betting_Phase
4. THE Game_Service SHALL include unit tests verifying that cash out is rejected outside Running_Phase
5. THE Game_Service SHALL include unit tests verifying monetary precision with various centavo amounts and Multiplier values
6. THE Game_Service SHALL include unit tests for the Provably_Fair_Algorithm verifying deterministic crash point generation
7. THE unit tests SHALL achieve at least 90% code coverage for the domain layer

### Requirement 24: Property-Based Testing for Provably Fair Algorithm

**User Story:** As a developer, I want property-based tests for the provably fair algorithm, so that I can verify correctness across a wide range of inputs.

#### Acceptance Criteria

1. THE Game_Service SHALL include property-based tests verifying that the same Server_Seed always produces the same Crash_Point
2. THE Game_Service SHALL include property-based tests verifying that different Server_Seeds produce different Crash_Points
3. THE Game_Service SHALL include property-based tests verifying that the Server_Seed_Hash matches the SHA-256 hash of the Server_Seed
4. THE Game_Service SHALL include property-based tests verifying that Crash_Point values are always greater than or equal to 1.00x
5. THE Game_Service SHALL include property-based tests verifying that the House_Edge is correctly applied to crash point calculations
6. THE property-based tests SHALL use fast-check library and run a minimum of 100 iterations per property

### Requirement 25: End-to-End Testing for Game Flow

**User Story:** As a developer, I want end-to-end tests for the complete game flow, so that I can verify the service works correctly in an integrated environment.

#### Acceptance Criteria

1. THE Game_Service SHALL include E2E tests for the complete game flow: betting phase → bet placement → round start → multiplier progression → cash out → round crash → new round
2. THE Game_Service SHALL include E2E tests verifying that bets are rejected outside the Betting_Phase
3. THE Game_Service SHALL include E2E tests verifying that cash outs are rejected outside the Running_Phase
4. THE Game_Service SHALL include E2E tests verifying that players cannot place multiple bets in the same Round
5. THE Game_Service SHALL include E2E tests verifying that WebSocket events are broadcast correctly to all connected clients
6. THE Game_Service SHALL include E2E tests verifying integration with the Message_Broker for Bet_Placed_Event, Cashout_Event, and Bet_Lost_Event
7. THE E2E tests SHALL use a test database separate from the development database
8. THE E2E tests SHALL use a test Message_Broker instance or mock

### Requirement 26: Betting Phase Timer Configuration

**User Story:** As a system administrator, I want to configure the betting phase duration, so that I can adjust game pacing based on player preferences.

#### Acceptance Criteria

1. THE Game_Service SHALL read the Betting_Phase duration from environment configuration
2. THE Betting_Phase duration SHALL be configurable in seconds with a default value of 10 seconds
3. THE Betting_Phase duration SHALL have a minimum value of 5 seconds and a maximum value of 60 seconds
4. WHEN the Betting_Phase duration is changed, THE Game_Service SHALL apply the new duration to newly created rounds
5. IF an invalid Betting_Phase duration is configured, THEN THE Game_Service SHALL log an error and use the default value

### Requirement 27: Bet Amount Validation

**User Story:** As the Game Service, I want to enforce bet amount limits, so that players cannot place bets outside the allowed range.

#### Acceptance Criteria

1. WHEN a bet placement request is received, THE Game_Service SHALL validate that the bet amount is at least 100 centavos (1.00)
2. WHEN a bet placement request is received, THE Game_Service SHALL validate that the bet amount is at most 100000 centavos (1000.00)
3. IF a bet amount is below the minimum, THEN THE Game_Service SHALL reject the bet with an error indicating the minimum bet amount
4. IF a bet amount is above the maximum, THEN THE Game_Service SHALL reject the bet with an error indicating the maximum bet amount
5. THE minimum and maximum bet amounts SHALL be configurable via environment variables

### Requirement 28: Round State Transition Validation

**User Story:** As the Game Service, I want to enforce valid round state transitions, so that the game state machine operates correctly.

#### Acceptance Criteria

1. THE Round entity SHALL only allow transitions from Betting_Phase to Running_Phase
2. THE Round entity SHALL only allow transitions from Running_Phase to Crashed state
3. THE Round entity SHALL only allow transitions from Crashed state to Finished state
4. IF an invalid state transition is attempted, THEN THE Round entity SHALL reject the transition and throw an error
5. THE Game_Service SHALL log all Round state transitions with timestamps and Round IDs

### Requirement 29: Bet State Transition Validation

**User Story:** As the Game Service, I want to enforce valid bet state transitions, so that bet lifecycle is managed correctly.

#### Acceptance Criteria

1. THE Bet entity SHALL only allow transitions from Pending to Active state
2. THE Bet entity SHALL only allow transitions from Active to CashedOut or Lost state
3. IF an invalid state transition is attempted, THEN THE Bet entity SHALL reject the transition and throw an error
4. THE Game_Service SHALL log all Bet state transitions with timestamps, Bet IDs, and player IDs

### Requirement 30: Hash Chain Integrity

**User Story:** As a player, I want the server seed hash chain to be verifiable, so that I can trust the game has not been manipulated.

#### Acceptance Criteria

1. WHEN a Round is created, THE Game_Service SHALL generate the next Server_Seed by hashing the previous Server_Seed
2. THE Game_Service SHALL store the hash chain linkage in the Round_Repository
3. WHEN a Round is finished, THE Game_Service SHALL reveal the Server_Seed which can be hashed to verify it matches the previously published Server_Seed_Hash
4. THE Game_Service SHALL provide an endpoint to retrieve the hash chain for verification
5. THE hash chain SHALL be unbroken from the first round to the current round
